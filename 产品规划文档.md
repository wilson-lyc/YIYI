# Mac 划词翻译工具产品规划文档

## 1. 产品概述

### 1.1 产品名称
YIYI

### 1.2 产品定位
一款原生开发（macOS Native）的划词翻译工具。用户在任意应用中选中文本后，可通过快捷键唤起翻译浮窗，快速获得目标语言译文。

### 1.3 目标用户
- 需要高频阅读英文/多语言内容的开发者、产品经理、研究人员
- 需要在文档、网页、PDF、IDE 等跨应用场景下即时翻译的用户
- 偏好轻量、低打扰、响应快的 macOS 原生工具用户

### 1.4 核心价值
- **跨应用可用**：在任意可选中文本场景可触发
- **极速体验**：划词后快速反馈，不打断阅读流
- **高质量翻译**：翻译能力由 LLM 提供，优先保证语义准确与上下文理解
- **可扩展架构**：当前支持 OpenAI 协议，后续可扩展更多模型服务

## 2. 功能规划

### 2.1 MVP（最小可行版本）功能边界
MVP 以“可用、稳定、可验证”为目标，仅覆盖以下核心能力：

1. 基础 UI（状态栏 + 翻译浮窗）
2. 划词文本获取
3. 通过 OpenAI 协议调用 LLM 获取翻译
4. 基础设置（API Key、模型、源/目标语言、快捷键）

### 2.2 功能模块拆分

#### A. 基础 UI 模块
- macOS 状态栏图标（常驻后台）
- 翻译浮窗（显示原文、译文、加载状态、错误提示）
- 首次引导（辅助开启辅助功能权限/输入监控权限）
- 设置页（最简版）

#### B. 划词检测与文本获取模块
- 全局快捷键触发翻译
- 从系统当前选中内容读取文本
- 文本预处理：去除多余空白、长度截断、特殊字符清理
- 失败兜底提示（未选中文本、权限不足、应用不支持）

#### C. LLM 翻译模块（OpenAI 协议）
- OpenAI 协议客户端封装（兼容 `base_url`、`api_key`、`model`）
- Prompt 模板（翻译任务，支持源/目标语言）
- 请求超时与重试策略
- 响应解析与错误映射（鉴权失败、限流、网络错误）

#### D. 基础设置模块
- API Key 输入与本地安全存储（Keychain）
- 模型名称配置（例如 `gpt-4o-mini` 等）
- 目标语言设置（默认中文）
- 快捷键设置（默认 `⌥ + D`，可调整）

### 2.3 暂不纳入首版（后续迭代）
- OCR 截图翻译
- 历史记录与收藏
- 术语表与翻译风格预设
- 多服务商智能路由
- 离线翻译模型

## 3. 技术规划

### 3.1 技术栈建议（原生 macOS）
- **开发语言**：Swift
- **UI 框架**：SwiftUI（配合 AppKit 处理系统级能力）
- **系统能力**：
  - 状态栏：`NSStatusBar`
  - 全局快捷键：`NSEvent`/第三方轻量封装
  - 辅助功能与选中文本获取：Accessibility API（AXUIElement）
- **网络层**：`URLSession`
- **本地存储**：
  - 配置：`UserDefaults`
  - 密钥：`Keychain`

### 3.2 架构设计
采用模块化 + 简化 MVVM：

- `UI`：状态栏、浮窗、设置页面
- `Capture`：选中文本捕获与预处理
- `Translation`：LLM 请求与结果解析
- `Config`：用户配置管理
- `Infrastructure`：日志、网络、错误处理

数据流：
1. 用户划词并触发快捷键
2. `Capture` 模块读取选中文本
3. `Translation` 模块构建 Prompt 并调用 OpenAI 协议接口
4. UI 展示译文或错误信息

### 3.3 OpenAI 协议适配策略
- 请求接口优先使用 Chat Completions（后续可扩展 Responses API）
- 配置参数：
  - `base_url`（默认 OpenAI 官方地址，可自定义）
  - `api_key`
  - `model`
  - `temperature`（默认低值以保证翻译稳定性）
- 通用消息模板：
  - system：约束翻译风格与输出格式
  - user：包含原文、源语言、目标语言

### 3.4 安全与隐私
- API Key 仅存储于系统 Keychain
- 默认不上传历史记录（首版无历史功能）
- 日志脱敏（不打印完整 API Key 和原文内容）

### 3.5 稳定性与可维护性
- 网络超时（例如 15s）+ 指数退避重试（有限次数）
- 统一错误码与错误文案
- 为核心流程预留单元测试点（文本预处理、响应解析、错误映射）

## 4. 阶段性开发计划

> 按要求采用分阶段开发：**先实现基础 UI**，再实现**划词获取文本**，最后实现**调用 LLM 翻译**。

### 阶段 1：基础 UI 搭建（Week 1）
**目标**：可运行的 macOS 原生应用骨架。

**交付内容**：
- 状态栏常驻图标与菜单
- 翻译浮窗（原文区/译文区/加载态占位）
- 设置页面基础结构（暂用 mock 数据）

**验收标准**：
- 应用可后台运行
- 点击状态栏菜单可打开浮窗与设置页
- UI 在深色/浅色模式下显示正常

### 阶段 2：划词文本获取能力（Week 2）
**目标**：拿到“用户当前选中文本”。

**交付内容**：
- 全局快捷键触发流程
- Accessibility 权限引导与检测
- 选中文本读取与预处理
- 未选中文本时的错误提示

**验收标准**：
- 在主流文本场景（浏览器、编辑器、文档）可稳定获取选中文本
- 未授权或读取失败时有清晰引导
- 文本长度与空白处理符合预期

### 阶段 3：接入 LLM 翻译（OpenAI 协议）（Week 3）
**目标**：完成“从选中文本到译文展示”的闭环。

**交付内容**：
- OpenAI 协议客户端（可配置 `base_url/api_key/model`）
- 翻译 Prompt 设计与响应解析
- 翻译结果渲染到浮窗
- 失败重试与错误提示

**验收标准**：
- 输入有效文本可返回可读译文
- API Key 错误、网络异常、限流等场景可识别并提示
- 平均响应时延满足可用性目标（如 1~3 秒，视模型与网络条件）

### 阶段 4：首版打磨与发布准备（Week 4）
**目标**：提升可用性并准备内部试用。

**交付内容**：
- 设置项完善（默认目标语言、快捷键配置、模型配置）
- 基础埋点与日志（本地）
- 回归测试与问题修复
- 打包与签名流程梳理（内部分发）

**验收标准**：
- 核心链路稳定运行
- 崩溃率与明显阻断问题可控
- 可交付给小范围用户试用

## 5. 里程碑与版本建议

- **v0.1.0（阶段1结束）**：仅 UI 骨架
- **v0.2.0（阶段2结束）**：UI + 划词获取
- **v0.3.0（阶段3结束）**：UI + 划词 + LLM 翻译闭环（MVP）
- **v1.0.0（阶段4结束）**：可内部试用的稳定版本

## 6. 风险与应对

1. **选中文本在部分应用不可读**
   - 应对：做应用兼容性清单，提供失败提示与备用触发方案
2. **LLM 调用成本与延迟波动**
   - 应对：默认轻量模型 + 可配置模型策略 + 请求超时控制
3. **权限导致初次使用门槛高**
   - 应对：首次引导流程可视化，提供一步步授权说明
4. **接口兼容性差异（OpenAI 协议实现差异）**
   - 应对：协议层抽象，严格校验请求/响应字段，增强错误兜底
